1. Test problem framing

OpenSpec change `migrate-marketing-products-personas` requires products/personas to work through shared domain/application/infrastructure seams, with SurrealDB repository tests using real `mem://`, provider calls mockable, and UI reusable/localized. The gap report `.pi/gaps/migrate-marketing-products-personas-implementation-gap.md` narrows this test plan to four fixes:

- `SurrealProductRepository.update()` must use merge semantics so `createdAt` survives updates.
- `ProductDetailView.svelte` must render valid created dates instead of `Invalid Date`.
- persona generation must return a friendly/configuration-safe error when the AI provider is not configured, without leaking raw provider/model internals.
- `ProductCard` must expose a navigable product name/link to the product detail route.

Primary regression risk: prior Surreal update semantics likely replaced the full record, dropping fields such as `createdAt`; downstream UI then received missing/invalid dates. Persona generation currently calls the AI gateway before the persistence rollback `try` block, so provider/config errors can escape directly. Product cards currently render the name as plain title text with only persona navigation as a button.

Test runner/conventions:

- Domain package tests use `bun:test` imports (`describe`, `test`, `expect`, `beforeEach`, `afterEach`) and are run directly with `bun test <path>`.
- DB-backed tests must use real SurrealDB `mem://`, unique namespace/database per `beforeEach`, and close the connection in `afterEach`; do not add in-memory repository fakes for database-backed ports.
- Application service tests currently use fake repositories and mock only the external AI gateway boundary; this is allowed for provider/config error tests.
- Desktop app unit tests use Vitest (`bun run --filter desktop-app test:unit` or targeted `cd apps/desktop-app && bunx vitest run --project unit <file>`).
- UI component/browser tests use Vitest browser + `vitest-browser-svelte` in files named `*.browser.test.ts` (`bun run --filter ui test:browser`). No marketing UI browser tests currently exist, so ProductCard/ProductDetailView likely need a new test fixture/file unless implementation relies on e2e only.

3. Proposed tests per layer with file paths and test descriptions

Repository/infrastructure layer

- Extend `packages/domain/src/infrastructure/database/marketing/product-persona-repository.test.ts` in `describe('SurrealProductRepository')` near the existing `update changes fields` test.
  - Test name: `update preserves createdAt and changes updatedAt`.
  - Arrange: create a product with `name`, `description`, `targetAudience`, `features`, and `benefits`. Capture `created.createdAt` and `created.updatedAt`.
  - Act: call `productRepo.update(created.id, { name: 'Updated', description: 'Updated description' })`.
  - Assert:
    - `updated.name` and `updated.description` changed.
    - `updated.createdAt === created.createdAt`.
    - `updated.updatedAt !== created.updatedAt` (or parse dates and assert updated is >= created; exact inequality can be flaky if same millisecond, so safest is use a tiny delay or assert returned `updatedAt` is a valid ISO string and not undefined; implementation can add `await Bun.sleep(1)` if needed).
    - `updated.features` and `updated.benefits` still equal original arrays to prove merge semantics preserve untouched fields.
    - `await productRepo.findById(created.id)` also has the preserved `createdAt`.
  - Negative guard: do not only assert `createdAt` is defined; compare to the original value so this catches accidental reset-on-update.

Application service layer: persona generation config error

- Extend `packages/domain/src/application/marketing/persona-service.test.ts`.
  - Test name: `generateForProduct maps AI provider configuration failures to a friendly error`.
  - Arrange: create a `failingAi` implementing `IMarketingTextGenerationGateway` where `generatePersonas` throws a configuration-style error. Prefer a domain-specific/config error if implementation adds one; otherwise throw `new Error('model: claude-3-5-haiku-20241022')` to lock in that raw model/provider detail is not surfaced.
  - Act/assert: `await expect(svc.generateForProduct(product, 2)).rejects.toThrow(<friendly message>)`.
  - Additional assertions:
    - `expect(personaRepo.items).toHaveLength(0)` so no partial persona is persisted when the gateway fails before generation.
    - If checking the thrown message manually, assert it does not include `model:`, `claude`, `anthropic`, or API key/env var names.
  - Expected friendly wording should be stabilized by implementation/spec owner. Suggested contract: message contains `Persona generation is unavailable` and `configure` (case-insensitive), not raw provider details.

- Keep existing `generateForProduct rolls back on mid-loop failure` test (lines 113-124). If implementation broadens error mapping, ensure this test still verifies persistence rollback and either still throws `generation failed` or is updated to the new friendly persistence failure wording.

Application service layer: product update timestamp guard

- Optional but useful extension to `packages/domain/src/application/marketing/product-service.test.ts` near `update changes product fields`.
  - Test name: `update preserves product createdAt from repository result`.
  - Arrange: create a product, capture `createdAt`, update name.
  - Assert `updated.createdAt === created.createdAt` and `updated.name === 'New'`.
  - Caveat: current fake repository already preserves `createdAt` by object spread, so this test mainly documents service contract; it will not catch the Surreal replace-vs-merge regression. The repository test above is mandatory.

UI component/browser layer: ProductDetailView date

- Add a new UI browser test file (no existing marketing UI test file was found): `packages/ui/src/lib/marketing/products/ProductDetailView.browser.test.ts` plus a small fixture component if needed.
  - Test name: `renders createdAt as a valid localized date for ISO timestamps`.
  - Render `ProductDetailView` with a product fixture whose `createdAt` is a known ISO string such as `2026-04-28T13:34:03.170Z` and `updatedAt` another valid ISO string.
  - Assert the rendered text includes `Created` and does not include `Invalid Date`.
  - For deterministic output, avoid asserting a locale-specific exact date unless the implementation uses a deterministic formatter. Prefer `not.toContain('Invalid Date')` plus existence of the product name and Created label.
  - If implementation adds a formatting helper, add a pure unit test for that helper as well (invalid/missing input returns a safe fallback such as `Unknown date`).

UI component/browser layer: ProductCard navigable link

- Add a new UI browser test file or combine with a product component browser test: `packages/ui/src/lib/marketing/products/ProductCard.browser.test.ts`.

5. Risks and fixtures needed

- SurrealDB timestamp comparison can be same-millisecond; if asserting `updatedAt` changes, insert a tiny delay before update or assert it is valid and not missing. The must-have assertion is exact preservation of `createdAt`.
- Use real SurrealDB `mem://` in repository tests; do not introduce an in-memory fake for `SurrealProductRepository` coverage.
- Application service tests may use fake repositories because the boundary under test is the AI gateway error mapping, not DB persistence.
- Friendly AI error wording needs a stable contract. Tests should verify no leakage of raw `model: claude-3-5-haiku-20241022`, provider names, API key names, or stack details.
- ProductDetailView date tests should not rely on a specific locale rendering unless the implementation makes formatting deterministic.
- ProductCard link tests need the final prop/route design. If `ProductCard` becomes reusable with a `href` prop, test the UI component directly. If routing stays app-local, test route composition instead.
- Svelte/browser tests may need small fixture components in `packages/ui/src/lib/marketing/products/__tests__` or adjacent to the component, following existing `vitest-browser-svelte` patterns.
